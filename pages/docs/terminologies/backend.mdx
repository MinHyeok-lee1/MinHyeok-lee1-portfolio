# 백엔드 용어

## HTTP (HyperText Transfer Protocol)

HTTP란?

1. HTTP는 웹에서 클라이언트와 서버 간에 데이터를 주고받기 위한 가장 기본적인 프로토콜이다.
2. 웹 브라우저(클라이언트)가 웹 서버에 요청을 보내고, 서버가 해당 요청에 응답하는 방식으로 작동한다.
3. HTTP는 상태를 저장하지 않는(stateless) 프로토콜로, 각 요청이 독립적으로 처리된다.

### 주요 특징

1. 메서드
   - HTTP는 GET, POST, PUT, DELETE, PATCH 등의 다양한 메서드를 사용하여 클라이언트가 서버에 특정 작업을 요청할 수 있다.
   - GET: 데이터를 요청하는 데 사용, 주로 웹 페이지를 요청하거나 데이터를 조회할 때 사용한다.
   - POST: 서버에 데이터를 제출하는 데 사용, 주로 폼 데이터 제출, 파일 업로드 등에 사용된다.
   - PUT: 서버에 데이터를 업데이트하는 데 사용된다.
   - DELETE: 서버에서 데이터를 삭제하는 데 사용된다.
   - PATCH: 부분적으로 데이터를 업데이트하는 데 사용된다.
2. URI (Uniform Resource Identifier)
   - 각 자원(resource)의 위치를 식별하기 위해 사용된다.
3. 헤더(Header)
   - 요청과 응답에 추가적인 정보를 포함할 수 있는 메타데이터이다.
   - 예: Content-Type, Authorization 등
4. 상태 코드(Status Code)
   - 서버 응답의 상태를 나타내는 코드이다.
   - 예: 200 OK, 404 Not Found, 500 Internal Server Error.

### HTTPS

HTTPS (HyperText Transfer Protocol Secure)란?

- HTTP의 보안 확장으로, 데이터 전송을 암호화하여 중간에서의 도청이나 변조를 방지한다.
- HTTPS는 SSL/TLS(보안 소켓 레이어/전송 계층 보안) 프로토콜을 사용하여 데이터를 암호화한다.
  - SSL/TLS: SSL은 원래 개발된 프로토콜이고, 현재는 TLS가 그 역할을 이어받고 있다.
  - 데이터 전송 시 암호화를 제공하며, HTTPS를 통해 암호화된 데이터를 안전하게 주고받을 수 있다.

## REST (Representational State Transfer)

REST란?

1. REST는 웹 서비스 설계 아키텍처로, HTTP 프로토콜을 기반으로 자원(Resource)을 정의하고 이를 조작하는 방식이다.
2. RESTful 서비스는 클라이언트와 서버 간의 통신을 단순화하며, 상태를 서버에 저장하지 않고 클라이언트가 전적으로 관리한다.

### 주요 특징

1. 자원 기반 구조(Resource-Based): URI를 통해 자원을 식별한다.
2. 무상태성(Statelessness): 각 요청은 독립적이며, 서버는 이전 요청에 대한 정보를 유지하지 않는다.
3. 캐시 가능(Cacheable): 응답 데이터는 캐시될 수 있으며, 이를 통해 네트워크 트래픽을 줄이고 응답 시간을 단축할 수 있다.
4. 계층형 시스템(Layered System): 클라이언트는 서버와 직접 통신하지 않을 수 있으며, 프록시나 게이트웨이를 통해 통신할 수 있다.

### 파생 기술

1. GraphQL
   - REST의 대안으로, 클라이언트가 정확히 필요한 데이터만 요청할 수 있도록 하는 쿼리 언어이다.
   - REST가 리소스 단위로 데이터를 가져오는 반면, GraphQL은 특정 필드를 요청할 수 있다.
2. gRPC
   - 구글에서 개발한 원격 프로시저 호출(RPC) 시스템으로, REST와 달리 바이너리 형식으로 데이터를 전송하여 더 효율적이다.
   - 주로 내부 마이크로서비스 간 통신에 사용된다.

## API (Application Programming Interface)

API란?

1. API는 애플리케이션 간의 상호작용을 가능하게 하는 인터페이스이다.
2. 백엔드에서는 주로 서버가 클라이언트에게 데이터를 제공하기 위한 인터페이스를 API로 구현한다.

### 주요 특징

1. RESTful API
   - REST 아키텍처 스타일을 따르는 API이다.
   - URI, HTTP 메서드, 상태 코드 등을 활용하여 자원에 대한 작업을 정의한다.
2. SOAP (Simple Object Access Protocol)
   - XML을 사용하여 메시지를 교환하는 프로토콜이다.
   - REST보다 더 엄격하고 복잡하며, 보안이 중요한 엔터프라이즈 환경에서 주로 사용된다.
3. Web API
   - 웹을 통해 제공되는 API로, RESTful API, GraphQL, gRPC 등이 포함된다.

## Database (DB)

Database란?

1. 데이터베이스는 데이터를 체계적으로 저장하고 관리하는 시스템이다.
2. 백엔드에서 가장 중요한 역할 중 하나를 담당하며, 데이터의 무결성과 일관성을 유지하는 것이 중요한다.

### 주요 특징

1. 관계형 데이터베이스(SQL)
   - 테이블 간의 관계를 기반으로 데이터를 저장하며, SQL(Structured Query Language)을 사용하여 데이터를 조작한다.
   - 대표적인 예로 MySQL, PostgreSQL, Oracle 등이 있다.
   - 특징: 트랜잭션을 지원하며, 데이터 무결성 제약 조건(제약, 키)을 통해 일관성을 유지한다.
2. 비관계형 데이터베이스(NoSQL)
   - 테이블 구조가 아닌 다양한 형식(문서, 키-값, 컬럼 패밀리, 그래프)으로 데이터를 저장한다.
   - 대표적인 예로 MongoDB, Cassandra, Redis 등이 있다.
   - 특징: 데이터 모델링의 유연성이 높고, 대규모 분산 시스템에서 성능이 뛰어나다.

## ORM (Object-Relational Mapping)

ORM이란?

1. ORM은 객체 지향 프로그래밍 언어에서 데이터베이스의 데이터를 객체로 매핑하는 기술이다.
2. 개발자가 SQL 쿼리를 직접 작성하지 않고도 데이터베이스와 상호작용할 수 있도록 도와준다.

### 주요 특징

1. 코드 재사용성: 데이터베이스와의 상호작용을 추상화하여 코드 재사용성을 높인다.
2. 데이터베이스 독립성: ORM을 사용하면 코드 변경 없이 데이터베이스를 교체할 수 있다.
3. 자동화된 데이터베이스 작업: ORM은 데이터베이스 테이블을 자동으로 생성하고, 객체와 테이블 간의 매핑을 관리한다.

### 파생 기술

1. Hibernate: Java에서 가장 널리 사용되는 ORM 프레임워크로, JPA(Java Persistence API) 표준을 구현한다.
2. SQLAlchemy: Python에서 사용되는 ORM 라이브러리로, SQLAlchemy Core와 ORM을 제공하여 유연한 데이터베이스 작업을 지원한다.
3. Django ORM: Django 프레임워크에 내장된 ORM으로, Python 코드만으로 데이터베이스 작업을 수행할 수 있다.

## MVC (Model-View-Controller)

MVC란?

1. MVC는 소프트웨어 설계 패턴이다.
2. 애플리케이션을 세 가지 주요 컴포넌트로 분리하여 유지 보수성과 확장성을 높인다.

### 주요 특징

1. Model
   - 데이터와 비즈니스 로직을 처리하는 부분이다.
   - 데이터베이스와 상호작용하며 애플리케이션의 핵심 데이터를 관리한다.
2. View
   - 사용자에게 데이터를 표시하는 부분이다.
   - UI(User Interface)를 구성한다.
3. Controller
   - 사용자 입력을 처리하고 모델과 뷰를 조정하는 부분이다.
   - 사용자의 요청을 받아 적절한 모델을 호출하고, 결과를 뷰에 전달한다.

### 파생 개념

1. MVVM (Model-View-ViewModel)
   - 주로 프론트엔드에서 사용되는 패턴이다.
   - ViewModel을 추가하여 뷰와 모델 간의 데이터를 관리한다.
   - 주로 Angular, React와 같은 프레임워크에서 사용된다.
2. MVP (Model-View-Presenter)
   - 뷰와 모델 간의 직접적인 상호작용을 제거한다.
   - 프레젠터가 모든 비즈니스 로직을 처리하도록 하는 패턴이다.

## Session & Cookie

세션과 쿠키는 웹 애플리케이션에서 사용자를 식별하고 상태를 유지하기 위한 방법이다.

### Session

Session이란?

1. 서버에 사용자의 상태 정보를 저장한다.
2. 사용자는 고유한 세션 ID를 통해 서버에서 본인의 상태를 식별할 수 있다.
3. 이 세션 ID는 보통 쿠키에 저장된다.

#### Session의 장단점

1. 장점: 서버에서 관리되므로 클라이언트 측에서 정보가 노출되지 않으며, 보안성이 높다.
2. 단점: 서버에 부하가 걸릴 수 있으며, 확장성이 제한될 수 있다.

### Cookie

Cookie란?

1. 클라이언트의 브라우저에 저장되는 작은 데이터로, 사용자가 웹 사이트를 방문할 때 정보를 기억하는 데 사용된다.

#### Cookie의 장단점

1. 장점: 클라이언트 측에서 관리되므로 서버 부하가 적다.
2. 단점: 클라이언트가 쿠키를 수정하거나 삭제할 수 있으며, 보안에 취약할 수 있다.

### 파생 개념

1. JWT (JSON Web Token)
   - JWT는 세션과 쿠키의 대안으로, 클라이언트 측에서 상태를 유지할 수 있는 방법이다.
   - JWT는 클라이언트와 서버 간의 인증 및 정보를 안전하게 교환하기 위해 사용된다.

## JWT (JSON Web Token)

JWT란?

1. JWT는 JSON 형식의 데이터를 이용한 인증 토큰으로, 서버와 클라이언트 간의 인증 및 정보 교환을 위한 방법이다.
2. JWT는 세 부분(헤더, 페이로드, 서명)으로 구성되어 있으며, 주로 인증 및 권한 부여에 사용된다.

### 주요 특징

1. 자기 포함형 토큰(Self-Contained Token)
   - JWT는 필요한 모든 정보를 자체적으로 포함하고 있다.
   - 따라서 서버가 상태를 저장할 필요가 없다.
2. 암호화 및 서명
   - JWT는 기본적으로 서명되어 있어, 토큰의 무결성을 확인할 수 있다.
   - 민감한 정보가 포함될 경우 암호화할 수도 있다.
3. 짧은 유효 기간
   - 보안을 위해 JWT는 일반적으로 짧은 유효 기간을 가진다.
   - 유효 기간이 만료되면 토큰을 재발급해야 한다.

### 파생 개념

1. OAuth: OAuth는 타사 서비스에 인증 및 권한을 위임할 수 있는 표준 프로토콜로, JWT를 사용하여 액세스 토큰을 발급하고 관리할 수 있다.
2. OpenID Connect: OAuth 2.0 위에 구축된 인증 프로토콜로, 사용자 인증에 중점을 둔다.

## WebSocket

WebSocket이란?

1. WebSocket은 클라이언트와 서버 간의 양방향 통신을 가능하게 하는 프로토콜이다.
2. 실시간 데이터 전송이 필요한 애플리케이션에서 많이 사용된다.

### 주요 특징

1. 실시간 통신
   - HTTP는 요청-응답 패턴으로 작동한다.
   - WebSocket은 지속적인 연결을 유지하면서 양방향 통신을 할 수 있다.
2. 이벤트 기반
   - WebSocket은 이벤트를 기반으로 메시지를 주고받을 수 있다.
   - 서버가 클라이언트에 데이터를 푸시할 수 있다.
3. 낮은 오버헤드
   - WebSocket 연결은 지속적이므로, HTTP보다 적은 오버헤드로 더 많은 데이터를 효율적으로 전송할 수 있다.

### 파생 기술

1. Socket.IO: WebSocket을 포함하여 다양한 실시간 통신 프로토콜을 지원하는 라이브러리로, 주로 Node.js 환경에서 사용된다.
2. STOMP: 메시지 지향 미들웨어와 함께 사용되는 단순 텍스트 기반의 프로토콜로, WebSocket 위에서 실행될 수 있다.

## Load Balancing (로드 밸런싱)

Load Balancing이란?

1. 로드 밸런싱은 여러 서버에 트래픽을 고르게 분산하여 시스템의 안정성, 성능, 확장성을 높이는 기술이다.

### 주요 특징

1. 트래픽 분산: 로드 밸런서는 들어오는 요청을 여러 서버에 분산하여 서버 과부하를 방지한다.
2. 고가용성: 특정 서버에 장애가 발생해도 다른 서버가 요청을 처리할 수 있도록 하여 서비스 중단을 최소화한다.
3. 수평 확장성: 로드 밸런서를 통해 서버를 추가하여 시스템 성능을 쉽게 확장할 수 있다.

### 파생 개념

1. DNS 로드 밸런싱
   - DNS 레벨에서 트래픽을 분산하는 방법으로, 여러 IP 주소를 하나의 도메인에 매핑하여 클라이언트 요청을 분산한다.
2. 리버스 프록시
   - 로드 밸런서가 리버스 프록시 역할을 하여 클라이언트 요청을 서버로 전달하고, 서버 응답을 클라이언트로 반환한다.
   - Nginx, HAProxy 등이 대표적인 리버스 프록시이다.

## Caching (캐싱)

Caching이란?

1. 캐싱은 자주 요청되는 데이터를 임시 저장소(캐시)에 저장하여 요청에 대한 응답 속도를 높이는 기술이다.
2. 서버 측 캐싱과 클라이언트 측 캐싱이 있으며, 성능 최적화에 매우 중요하다.

### 주요 특징

1. 서버 측 캐싱
   - 서버가 자주 요청되는 데이터를 메모리에 저장하여 빠르게 응답할 수 있도록 한다.
   - 예: Redis, Memcached.
2. 클라이언트 측 캐싱
   - 브라우저나 클라이언트 애플리케이션이 캐시된 데이터를 사용하여 서버 요청을 줄이고 응답 시간을 단축한다.
3. TTL (Time To Live)
   - 캐시 데이터의 유효 기간을 설정하여, 일정 시간이 지나면 캐시가 무효화되고 새롭게 데이터를 가져온다.

### 파생 개념

1. CDN (Content Delivery Network)
   - 전 세계 여러 위치에 캐시 서버를 두어 사용자에게 가까운 서버에서 콘텐츠를 제공하여 응답 속도를 높인다.
2. Lazy Loading
   - 필요한 시점에 데이터를 로드하여 초기 로딩 시간을 줄이는 기법이다.
   - 주로 이미지나 비동기 데이터 요청에 사용된다.

## CI/CD (Continuous Integration/Continuous Deployment)

CI/CD란?

1. CI/CD는 개발과 배포를 자동화하여 개발 속도를 높이고 코드 품질을 유지하는 방법론이다.

### Continuous Integration (CI)

1. Continuous Integration란?
   - 개발자들이 작성한 코드를 지속적으로 병합하고 테스트하는 과정이다.
   - 코드 병합 시 자동으로 빌드 및 테스트가 실행되어 코드의 품질을 보장한다.

### Continuous Deployment (CD)

1. Continuous Deployment란?
   - 검증된 코드를 자동으로 배포하는 과정이다.
   - 배포가 자동화되면, 새로운 기능이나 수정 사항을 신속하게 사용자에게 제공할 수 있다.

### 파생 개념

1. Continuous Delivery
   - 코드 변경이 검증되고 프로덕션 환경으로 배포할 준비가 되었을 때까지 자동화된 방식으로 전달하는 프로세스이다.
   - Continuous Deployment와의 차이점은 최종 배포 단계가 자동화되지 않고 수동으로 진행될 수 있다는 점이다.
2. DevOps
   - CI/CD의 일환으로, 개발(Development)과 운영(Operations)을 통합하여 소프트웨어 개발 주기를 단축하고 품질을 높이는 접근 방식이다.
